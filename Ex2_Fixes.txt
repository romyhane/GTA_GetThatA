Fix: Support Any Number of Rooms (Files)
----------------------------------------
Problem in Ex2: 
The game logic was previously hardcoded to load specifically "adv-world_01.screen", "_02", and "_03".
It could not handle a variable number of level files.

Fix in Ex3: 
1. Dynamic File Scanning: 
   We utilized the C++ <filesystem> library to dynamically scan the working directory 
   for all files matching the pattern "adv-world_*.screen".
   
2. Vector Storage: 
   These files are stored in a vector and sorted alphabetically. The game now iterates through this vector, 
   supporting any number of levels (1 to N) automatically.

3. Dynamic Dark Room Detection:
   We removed the hardcoded coordinates for the "Dark Room" (Level 3). The loader now automatically detects 
   the Dark Floor characters ('~') in any map file and calculates the dark zone boundaries dynamically.



Fix: Decoupled Riddles from Level Index
---------------------------------------
Problem in Ex2: Riddles were hardcoded to match specific level numbers in 'riddles.txt'. 
Adding a '?' tile to a level without a matching header in the text file caused the riddle to be inactive.

Fix in Ex3: 
- Changed riddle loading logic to ignore numeric headers and load ALL riddles into a global pool.
- Implemented cyclic assignment: '?' tiles now receive questions from the pool sequentially (modulo size).
- This ensures that every riddle tile triggers a valid question regardless of the level number.



Fix: Dynamic Switch-Door Connection (Spatial Coupling)
----------------------------------------------------
Problem in Ex2: Switch-to-Door connections were hardcoded by coordinates in the code. 
Moving objects in the screen file resulted in broken logic (switches not opening anything).

Fix in Ex3:
- Implemented a "Spatial Coupling" algorithm. Upon loading a level, each switch automatically scans 
  and links to the *nearest* Door object (Euclidean distance).
- This allows level designers to place switches and doors freely in the text file without modifying the C++ code.



Fix: Spring Mechanics (Stay Key Release)
----------------------------------------
Problem in Ex2: Springs launched the player immediately upon contact, allowing no control or preparation time. 
The requirement to use the 'STAY' key for release was missing.

Fix in Ex3:
- Modified Spring logic to "Prime" the spring upon contact (compress animation only).
- The player now enters a "Locked" state on the spring.
- Launch is triggered ONLY when the player presses their designated STAY key ('s' for Romy, 'k' for Tomer).



Appeal regarding Deduction: "No way to configure keys assigned to each door"
---------------------------------------------------------------------------
We respectfully appeal the deduction regarding the lack of specific key-to-door configuration.
The assignment instructions explicitly state: 
"Each key may match one or more doors, depending on the game design."

Based on this freedom, we designed a "Generic Key" system (arcade style), where any collected key opens the nearest 
locked door. This was a conscious design choice, not a missing feature. 
Since our design does not require specific key-door matching (Color Coding), implementing a configuration mechanism 
for it would have been redundant and contrary to our chosen gameplay logic (YAGNI principle).
We have clarified this in the documentation for Ex3.



Appeal regarding Deduction: "Cannot change door destination via screen file"
----------------------------------------------------------------------------
We respectfully appeal this deduction. The grader assumed that doors are intended to function as "Portals" or "Teleporters" 
linking to specific files based on their ID.
However, in our game design, numbered doors (1-9) function strictly as **Locked Barriers / Gates** within the current room. 
They impede movement until removed by a key or switch.
They are NOT transport mechanisms.
Therefore, changing a door's number is not supposed to change a "destination" (since there is none), but rather just the identity of the barrier.
The progression between levels is linear and triggered by reaching the screen boundary, not by entering a specific door number.
We have updated the documentation to explicitly state that doors are barriers, not portals.



Clarification: Switches & Disappearing Doors
--------------------------------------------
Grader Comment: "(-1) switches cause doors to disappear".
Response: This is a conscious Design Choice. In our game, opening a door removes the barrier entirely to clear the path. 
We have updated the documentation to reflect this behavior explicitly.



Clarification: Co-op Obstacle Pushing
-------------------------------------
Grader Comment: "(-1) obstacle of size 2 or more cannot be moved by... combined strength".
Response: This limitation stems from the specific physics implementation.
Our Co-op system requires players to push "Side-by-Side" (adjacent to each other and the obstacle) to sum their forces effectively.
"Train" formations (Player A pushing Player B pushing Obstacle) are not supported by the current physics engine. 
We have added this to the "Game Instructions" to guide players on proper co-op techniques.
